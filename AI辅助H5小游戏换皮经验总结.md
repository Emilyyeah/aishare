# AI 辅助 H5 小游戏换皮：经验教训与方法论

> 背景：把开源水果合成游戏（Matter.js 物理引擎）换成斗罗 IP 皮肤，接入公司内部服务端接口。
> 本文聚焦**踩过的坑和真实教训**，不写正确的废话。

## 一、接口接入（顺利）

### 做对了什么

**先列差异表，再动手改代码。**

拿到 v1 和 v3 两份接口文档，第一步不是打开代码，而是逐字段比对，输出一张表：

| 维度 | v3 | v1 |
|------|----|----|
| version 值 | `v3` | `v1` |
| 独有请求字段 | `consume_revive_item_count`、`consume_double_item_count` | 无 |
| 独有返回字段 | `actual_score`、`is_limit`、道具余量 | 只有 `remain_challenge_count`、`biz_no` |

改动清单确定后只有 3 处，改完做残留检测（grep 所有 v3 相关字段），验证干净。

**教训：接口切换不难，难的是"漏改"。先有清单，再动手，最后验收。**

## 二、换皮（反复折腾）

这部分才是真正的教训来源。

---

### 教训 1：没搞清楚渲染架构就动手，是最贵的错误

这个游戏用 Matter.js 物理引擎渲染，存在两个独立的渲染层：

- **Matter.js Canvas**：物理世界，有自己的逻辑坐标系（640×700），球、墙、菜单背景图都在这里
- **HTML DOM**：普通的网页元素，用 CSS 定位

菜单页的标题、金框、开始按钮最初全部放在 Matter.js Canvas 里——这直接导致：

- 调金框的 y 坐标 → 标题和按钮位置全乱
- 开始按钮超出画布 700px 底边 → 被截断
- 画布背景色和外部背景图有色差 → 出现"底纹"

**根因：展示元素（标题、装饰框）放进了有物理约束的引擎里，本质上是用错了工具。**

正确做法：**菜单页的纯展示元素（标题、金框合成图、开始按钮）全部放 HTML DOM 层，用 CSS 绝对定位，和物理引擎彻底解耦。** 三块互不影响，改哪块动哪块。

---

### 教训 2：没有提前建立坐标系映射，反复盲调

Matter.js 逻辑坐标是 640×700，但 HTML 页面实际显示尺寸因设备不同而变化（手机 430×932，桌面浏览器 813px 高）。

**我做错的事：** 直接硬编码逻辑坐标（比如 `y=520`），却期望它在手机和桌面上都对齐背景图的"尖尖"。这根本不可能，因为背景图是 CSS `cover` 模式铺在整个页面上的，它的位置和 Matter.js 的坐标系根本不在同一套映射关系里。

**正确做法：**
1. 先搞清楚背景图在目标设备（手机）上的实际渲染位置
2. 把要对齐的元素放 HTML 层，用 `position: absolute` + 百分比定位
3. 逻辑坐标只用来控制物理游戏本身（球、地板、墙），不用来做视觉对齐

**结论：Matter.js 坐标系只管游戏逻辑，所有视觉对齐的事交给 CSS。**

---

### 教训 3：等比缩放是刚性约束，开始就要规范

多次出现金框变形（圆变椭圆）的问题，原因是 `xScale ≠ yScale`。

```js
// 错误——等比例不同，图片被拉伸
render: { sprite: { texture: 'bg.png', xScale: 540/640, yScale: 500/700 } }

// 正确——统一系数，不变形
const s = 0.75;
render: { sprite: { texture: 'bg.png', xScale: s, yScale: s } }
```

**这不是计算问题，是工作规范问题。** 一开始就应该定死：所有 Matter.js Sprite 贴图，必须使用统一缩放系数，禁止 x 和 y 单独设值。当时没有规范，每次改坐标顺手就把 scale 也动了，导致反复变形。

---

### 教训 4：Matter.js 透明背景有坑，要提前验证

设置 `background: 'transparent'` 后，Matter.js 每帧不会自动清屏（它原本靠填充背景色来清屏），导致帧叠加出现残影。

解法是手动注册 `beforeRender` 事件做 `clearRect`：

```js
Events.on(render, 'beforeRender', function() {
  render.context.clearRect(0, 0, render.options.width, render.options.height);
});
```

**教训：用第三方引擎的非常规用法（透明背景），要先看文档或做 POC 验证，不要边改边猜。**

---

### 教训 5：切图验收要在动手之前做，不是动手之后

换皮开始前，应该逐张切图确认：
- 尺寸是多少？
- 是否有透明通道？
- 背景是透明还是实色？（`bg-menu640×700.png` 自带蓝色底色，这直接影响透明背景方案）
- 是否有我没有的素材（比如 circle0 新切图里没有，但代码里用到了）

**实际情况是：** 先改代码，改到一半发现素材有底色、有素材缺失，再回来处理，浪费了大量时间。

**正确流程：**
```
收到切图 → 逐张验收（尺寸/透明/内容） → 建立素材清单 → 确认无缺漏 → 开始改代码
```

## 三、代码安全（正确但可以更早做）

推 GitHub 前做脱敏是必须的，但脱敏脚本应该在第一次推之前就写好，而不是被 push 失败倒逼出来。

**部署脚本的关键设计：用 `trap EXIT` 保证还原，而不是放在 push 后面。**

```bash
restore_files() {
  for f in "${FILES_TO_SANITIZE[@]}"; do
    [ -f "$f.__bak__" ] && mv "$f.__bak__" "$f"
  done
}
trap restore_files EXIT  # 无论成功失败都执行还原
```

早期版本是 `set -e` + 把还原放在 push 之后，push 失败就不还原了，本地文件停在脱敏状态。

## 四、方法论提炼

### 1. 分层原则：先理解架构，再决定在哪一层改

每个项目都有分层：物理引擎层、渲染层、DOM 层、CSS 层。改任何东西之前，先确认它属于哪一层，以及该层的约束是什么。**改错了层是最贵的错误，会让后续所有工作都在补洞。**

### 2. 解耦原则：不同职责的元素放不同的层

- 游戏物理逻辑（球的碰撞、重力、投放）→ Matter.js
- 视觉装饰（标题、背景框、按钮）→ HTML + CSS
- 两者之间的信息传递 → 最小化，只传必要的数据

把视觉装饰放进物理引擎，是反模式。

### 3. 先量图再写坐标，不要边猜边改

有参考效果图时，先在图上量出关键元素的位置比例，换算成代码坐标，一次写对。不要写一个数，截图看一眼，再改，再截图——这是最低效的工作方式，消耗大量时间和 token。

**量图方法：**
- 确定参考图的总高度
- 目标元素中心在总高度的百分比
- 换算到逻辑坐标 = 百分比 × 画布高度

### 4. 规范先行：约定好再动手

换皮前应该先定好：
- 所有 Sprite 缩放只用统一系数
- 展示元素一律放 HTML DOM
- 素材引用路径命名规则
- 画布的坐标原点和尺寸边界

规范没有先定，事后靠 review 补，成本翻倍。

### 5. 把可复用的过程沉淀为工具

- 脱敏推送脚本 → 每个项目都能用
- 接口差异分析模板 → 下次接口切版直接套
- 切图验收清单 → 下次换皮直接用

**AI 帮你做完一件事之后，问自己：这件事下次还会发生吗？如果是，就让 AI 帮你把流程固化下来。**

### 教训 6：sprite 缩放系数错误导致球体视觉间隙

**现象：** 球体之间有明显间隙，物理上已经紧贴但视觉上像隔着空气。

**根因：** 切图素材 1024×1024，但角色头像主圆只占约 82%（直径 843px），四周有透明留白。代码用 `radius / 512` 做 sprite 缩放，意味着物理碰撞体边界 = 图片边界，但图片里有 18% 是透明的，所以视觉球比碰撞体小一圈。

**修复：** 通过 Python 测量主圆实际占比（843/1024 ≈ 82%），将缩放系数从 `/ 512` 改为 `/ 422`（512 × 82% ≈ 422），放大约 21%，让可见圆形填满碰撞体。

**教训：** 换皮前先测量素材中实际内容的占比，不要假设内容填满整张图。一行 Python 就能算清楚，比盲猜快 10 倍。

```python
from PIL import Image
img = Image.open('circle1.png').convert('RGBA')
# 水平扫描中心行找主圆边界
# 主圆直径 / 图片宽度 = 实际占比
```

### 教训 7：游戏逻辑空间和背景素材比例不匹配，反复试错

**现象：** 游戏背景框内的金色边线和球体落地点、死亡线完全对不上，球悬空在底线上方。

**根本矛盾：** Matter.js 画布 640×700（比例 0.914），缩放到屏幕后 CSS 高度只有 470px，但游戏区域背景（含金色边框）设计高度需要 742px。两者比例不一致，必然有一个会错位。

**走过的弯路（不要再重复）：**
- 试过 CSS marginTop / marginBottom 偏移 → 不同屏幕尺寸会漂移
- 试过 render.bounds 视口偏移 → 底部落地对不上
- 试过 overflow: hidden 裁切 → 球被截断
- 试过 z-index 层级方案 → 素材全不透明，挡住球

**正确解法：** 让美术把游戏逻辑区单独切图（只含金色框内区域），尺寸和画布 CSS 比例一致（860×940 @2x = 430×470 @1x = 画布 CSS）。背景图和画布 1:1 绑定，不受 CSS 布局影响，天然对齐。

**同时：** 将 `Game.height` 从 700 增大到 800，顶部多出 100px 作为丢球空间，底部地板固定到金色下横线位置（y=790），死亡线 = 790 - 616 = 174（保持可掉落高度 616 不变）。

```
y=0   ── canvas 顶部
y=100 ── 丢球位置 (previewBallHeight)
y=174 ── 死亡线 (loseHeight)
          可掉落高度 616
y=790 ── 地板 = 金色下横线
y=800 ── canvas 底部
```

**教训：比例问题只有两个出路：改画布比例，或改素材比例。CSS hack 全是治标不治本，每换一个屏幕尺寸就崩一次。让美术按画布比例出图才是根本解。**

---

### 教训 8：特效溢出边框——固定尺寸 vs 容器尺寸

**现象：** 大合成特效（放射光线）左侧超出游戏边框。

**根因：** `#big-merge-rays` 写死了 `width: 500px; height: 500px`，但画布 CSS 宽度只有 430px，500px 自然溢出。

**修复：** 改为 `width: 100%; aspect-ratio: 1`，跟随容器宽度，保持正方形不变形。父容器加 `overflow: hidden` 裁切溢出部分。

**教训：** 特效元素尺寸用百分比，不要写死像素值。写死像素 = 对某一个设备尺寸精确，对所有其他设备不对。

---

### 教训 9：只改一处却动了无关代码

**多次发生：** 被要求"只改某一处"，却顺手优化了周边代码——改顶栏字号、按钮尺寸、调整 padding，导致用户反复喊"还原还原"。

**正确执行方式：**
1. 确认用户说的是哪个元素
2. 只改该元素，绝不"顺便"动其他东西
3. 有疑问先问，不要靠猜

### 方法论补充

**6. 让美术按游戏坐标系出图，而不是让代码去适配美术图**

如果游戏画布是 640×800，就让美术把游戏区域图出成 640×800 比例（@2x 就是 1280×1600）。不要拿一张 860×1864 的全屏图然后靠代码去裁切、偏移、对齐——这条路走不通，在不同屏幕尺寸上必然错位。

**7. 遇到"反复试改同一个问题"，立刻停下来想根因**

连续试 3 次以上还没解决，说明方向错了，不是参数没调对。正确做法：停下来，把问题拆解成"坐标系 A 里的位置 → 坐标系 B 里的位置 → 代码参数"，先在纸上算清楚，再改代码。

**8. AI 执行前要先对齐"成功标准"**

每次改动前先说清楚"改完后我期望看到什么"，而不是"帮我改一下 xxx"。前者 AI 可以验证并迭代，后者 AI 只能猜你想要什么然后试错。
